# 深入浅出目标跟踪：从底层数学神器到 DeepSORT 与 ByteTrack 核心源码深度剖析

在视频里跟踪多个目标，目前工业界最主流的玩法叫 **Tracking-by-Detection（基于检测的跟踪）**。
逻辑极其直接：

1. **找目标（Detection）**：先用 YOLO 在每一帧里画上框。
2. **连轨迹（Tracking）**：把上一帧和这一帧的框“连线”，同一个目标的框连在一起，就成了轨迹。

难点在于**“数据关联”**：当几个人交错而过，或者被遮挡时，怎么保证 ID 不标错？为了解决这个问题，算法工程师们往工具箱里塞进了许多精妙的数学工具。

## 第一部分：揭开黑盒——目标跟踪的“四大数学神器”

无论是经典的 DeepSORT，还是现代巅峰 ByteTrack，本质上都是对下面这四个底层数学工具的巧妙拼装。

1. **卡尔曼滤波 (Kalman Filter) —— 掌控“预测”与“不确定性”**
   * **作用**：根据过去预测现在，结合当前观测得出最靠谱的真实位置。
   * **通俗理解**：你开车进隧道（GPS断联），你会根据进隧道前的**速度和方向**在心里估算位置（**预测**）。出隧道时GPS恢复，显示你在A点，你心里估算在B点。综合考虑你对估算的自信度和GPS的误差，你折中得出最真实的位置C（**更新**）。卡尔曼滤波在代码里不仅维护框的坐标，还维护“速度”，以及代表不确定性的“协方差矩阵”。
   * 对卡尔曼滤波感兴趣的可以看这篇文章https://www.bzarg.com/p/how-a-kalman-filter-works-in-pictures/
2. **马氏距离 (Mahalanobis Distance) —— 科学衡量“空间距离”**
   * **作用**：衡量 YOLO 的“检测框”和卡尔曼滤波的“预测框”在空间上离得有多远。
   * 它比普通直线距离牛在哪？卡尔曼预测出的位置通常是一个**椭圆形的不确定区域**（比如行人的高度变化通常远小于宽度变化）。马氏距离把这个**方差（不确定性方向）**考虑了进去，是一种绝对符合物理运动规律的空间度量。
3. **余弦距离 (Cosine Distance) —— 衡量“相貌”的相似度**
   * **作用**：衡量两张切图里的目标“长得像不像”。
   * 用 ReID 网络把行人图片变成 128 维特征向量。如果算欧氏距离，人走进阴影变暗，距离就会拉大。而**余弦距离衡量的是向量在多维空间中的“夹角”**。夹角越小，不管光线明暗，算法都知道是同一个人。
4. **匈牙利算法 (Hungarian Algorithm) —— 解决“乱点鸳鸯谱”**
   * **作用**：面对 $N$ 个预测轨迹和 $M$ 个检测框，找出一个**全局总代价最小**的一对一匹配方案。
   * 就像相亲大会，男1选了最合适的女A，可能导致男2只能选完全不合适的女B，整体极度糟糕。匈牙利算法不贪心，它通过矩阵变换，能在数学上绝对保证找出**让所有人总代价最小的全局最优方案**。
   * 对匈牙利算法感兴趣的可以看这篇文章https://zhuanlan.zhihu.com/p/62981901

## 第二部分：引入“视觉记忆”的经典 —— DeepSORT

目标一旦被长时间遮挡，卡尔曼滤波预测的位置就会“飞走”，导致目标重新出现时被当成新的人。**DeepSORT 的核心哲学是：不仅要看目标“怎么走”，还要看目标“长什么样”。**

### 1. DeepSORT 核心运转流程图

![deepsort-cascade-flow](https://raw.githubusercontent.com/bigbro13/deep-learning-handbook/master/assets/images/ch09/deepsort-cascade-flow.png)

### 2. 图解展开：DeepSORT 的每一步到底在干什么？

**【步骤 1 & 2：准备与预测】**
* 1.输入新的图像：YOLO 目标检测出目标框。立刻把目标框从原图上**裁剪**下来，送入一个独立的卷积神经网络（ReID），提取出目标框 128 维的“相貌特征向量”。
* 2.此时卡尔曼滤波根据上一帧的记忆，预测目标框老轨迹在这一帧应该走到了什么坐标。

**【核心步骤：级联匹配 Cascade Matching】**（*这是 DeepSORT 封神的地方*）
为什么叫“级联”？你可以把它理解为**“VIP 优先通道”**。

* **按丢失时间（age）排序排队**：算法维护了一个变量 `age`（即 time_since_update）。如果一个目标上一帧还在，`age=1`；如果被遮挡了 5 帧，`age=6`。**级联匹配的核心逻辑是：写一个 for 循环，让 `age=1` 的轨迹优先去挑选检测框！** 因为刚刚还在眼前的目标，位置最准，最容易配对。那些丢失了很久的轨迹，预测位置已经飘得很离谱了，必须往后排。
* **计算代价与门控**：
    1. **算长相**：计算检测框特征和轨迹特征的**余弦距离**。长得越像，代价越小。
    2. **物理门控（Gating）**：拿出**马氏距离**。如果检测框离轨迹的预测位置实在太远了（超出了物理运动的极限范围），即使它们特征一模一样（可能是穿同款衣服的两个人），也强行把配对代价设为**无穷大 (Infinity)**！直接一票否决。
* **匈牙利分配**：把结合了上述逻辑的代价矩阵扔给匈牙利算法，算出当前层级（比如 `age=1` 的群体）的最优配对。没配完的检测框，留给 `age=2` 的轨迹继续挑，直到 `age=30`。

**【保底步骤：IoU 匹配】**

* 级联匹配完了，可能还有剩下的检测框没配上，或者有些刚刚建立还没确认的轨迹（`Tentative`状态）不允许参加级联。这时候，DeepSORT 退化到最原始的方法：纯粹比较两个框的重叠面积（IoU）。如果重叠度够高，就凑合凑合配对了。这其实是为了应对突发的光照剧变导致外观特征突然失效的情况。

**【状态更新】**（对应图 I, J, K）

* **特征库更新（Feature Gallery）**：如果轨迹 A 配对成功，不仅更新它的坐标，还会把新的 128 维特征存入轨迹 A 的特征库中。DeepSORT 最多为每个目标保留 **最近 100 帧** 的特征，这样即使行人转身，算法也能逐渐适应他的背面长相。
* 状态机流转：彻底找不到的目标（`age>30`），标记为 `Deleted`，释放内存。新来的无主检测框，标记为 `Tentative`，如果连续 3 帧都能配上，才转正为 `Confirmed`。

### 3.代码实现

```python
def matching_cascade(distance_metric, max_distance, cascade_depth, tracks, detections, track_indices=None, detection_indices=None):
    """
    级联匹配核心流程
    :param distance_metric: 结合了外观余弦距离和位置马氏距离的度量函数
    :param max_distance: 最大允许距离阈值
    :param cascade_depth: 级联深度（最大丢失帧数，默认比如30帧）
    """
    # 记录未匹配的检测框
    unmatched_detections = set(detection_indices)
    matches = []

    # 按照丢失的帧数(time_since_update)从小到大进行匹配
    # 这就是"级联"的体现：优先把检测框分给刚刚才看过的轨迹，丢失太久的轨迹往后排
    for level in range(cascade_depth):
        if not unmatched_detections:
            break  # 所有检测框都配上了，提前结束
            
        # 找出丢失帧数恰好等于当前层级 level 的所有轨迹
        track_indices_l = [
            k for k in track_indices
            if tracks[k].time_since_update == 1 + level
        ]
        if not track_indices_l:
            continue # 当前层级没有符合要求的轨迹，继续下一层

        # 1. 计算代价矩阵 (融合特征距离与马氏距离)
        # distance_metric 会计算特征的余弦距离，如果空间距离太离谱会将其设为无穷大
        cost_matrix = distance_metric(
            tracks, detections, track_indices_l, list(unmatched_detections))

        # 2. 匈牙利算法，类似SORT中的 linear_sum_assignment，这里封装成了 min_cost_matching
        match_l, _, unmatched_detections_l = min_cost_matching(
            distance_metric, max_distance, tracks, detections,
            track_indices_l, list(unmatched_detections), cost_matrix=cost_matrix)

        # 3. 更新结果
        matches.extend(match_l)
        unmatched_detections = set(unmatched_detections_l) # 剩下的留给下一层级继续配

    unmatched_tracks = list(set(track_indices) - set(k for k, _ in matches))
    return matches, unmatched_tracks, list(unmatched_detections)
```

## 第三部分：变废为宝的现代巅峰 —— ByteTrack

随着 YOLO 的不断升级，检测器越来越强。学者们发现：DeepSORT 虽然稳，但必须外挂一个提取特征的 ReID 网络，导致系统非常臃肿，速度很慢。

以往的算法（包括 DeepSORT）有一个极其浪费的习惯：为了防止误检（把垃圾桶认成人），会**直接删掉 YOLO 给出的低置信度检测框（比如 Score < 0.5）**。
**ByteTrack 的伟大洞察在于：YOLO 不是瞎子，那个低分框往往是被前面的人挡住了一半的腿或胳膊！** 丢弃它，轨迹必然断裂。ByteTrack 抛弃了“相貌特征”，纯靠运动和 IoU，用极其巧妙的**二次匹配 (BYTE)** 拯救了低分框。

### 1. ByteTrack 核心运转流程图

![bytetrack-two-stage-flow](https://raw.githubusercontent.com/bigbro13/deep-learning-handbook/master/assets/images/ch09/bytetrack-two-stage-flow.png)

### 2. 图解展开：BYTE 二次匹配为什么被称为“神来之笔”？

ByteTrack 的流程没有任何复杂的“特征提取”，速度快到飞起，我们逐字拆解它的智慧：

**【步骤 1：分流（The Sorting Hat）】**

* YOLO 输出了一堆框，ByteTrack 根据阈值（比如 0.5）毫不留情地把它们分成两波：
    * **高分框 `Det_High`**：非常有把握的，清晰可见的完整人体。
    * **低分框 `Det_Low`**：把握不大，可能只露了个头，或者运动模糊了。如果分数低于 0.1，那真的是背景，直接丢弃。

**【第一次匹配：只给清晰的人配对】**

* 拿出卡尔曼预测的**所有历史轨迹**，只和 **高分框 `Det_High`** 计算 IoU 交并比。
* 送入匈牙利算法匹配。
* **关键点来了**：匹配完后，必然会剩下一些**“没分到高分框的孤儿轨迹”**。在以前的算法里，这些轨迹就被宣告“目标消失”了。但在实际场景中，这个人大概率只是被前面的大哥挡住了！

**【第二次匹配：去垃圾堆里找宝藏】**

* ByteTrack 把刚才**没配上的孤儿轨迹**收集起来。
* 它把目光投向了一开始被扔在一边的 **低分框 `Det_Low`**。
* 再次计算 IoU 和进行匈牙利匹配。并且，这次**放宽了匹配阈值**（因为是被遮挡的目标，YOLO 框得可能没那么准，只要大概挨在一起就行）。
* **奇迹发生了**：那个因为被遮挡导致置信度只有 0.3 的检测框，刚好和这个由于遮挡而预测停留在原地的轨迹配在了一起！卡尔曼滤波利用这个低分框成功更新了状态，**目标 ID 完美保持连续，没有断开！**

**【状态判定与清理】**
经历了两次严密的筛选，结果就非常清晰了：

* **高分框里剩下的**：连历史轨迹都不要它，说明它是一个刚进入画面的新目标，马上为它新建一条轨迹。
* **低分框里剩下的**：孤儿轨迹都不要它，说明它彻彻底底就是一团长得像人的树叶（背景噪声），直接销毁。
* **连低分框都没配上的轨迹**：说明这个人可能真的走出画面了，标记为 `Lost`，如果连续 30 帧都找不到，再彻底删除。

### 重要部分代码实现

```python
def update(self, output_results):
    """
    output_results: YOLO当前帧的输出 [x1, y1, x2, y2, score, class]
    """
    # 1. 将检测框拆分为高分和低分
    det_high = [det for det in output_results if det.score >= high_thresh]
    det_low  = [det for det in output_results if low_thresh <= det.score < high_thresh]
    
    # 获取当前所有需要跟踪的轨迹
    tracks = self.tracked_stracks
    
    # --- 【第一次关联】: 使用高分框 ---
    # 计算预测轨迹和高分检测框的 IoU 距离矩阵
    dists = iou_distance(tracks, det_high)
    # 使用匈牙利算法匹配
    matches_1, unmatched_tracks_1, unmatched_det_high = linear_assignment(dists, thresh=0.2)
    
    # 更新匹配成功的轨迹卡尔曼状态
    for trk_idx, det_idx in matches_1:
        tracks[trk_idx].update(det_high[det_idx])

    # --- 【第二次关联】: 使用低分框拯救丢失的轨迹 ---
    # 提取第一次没有匹配上的轨迹（候选可能是被遮挡了）
    r_tracked_stracks = [tracks[i] for i in unmatched_tracks_1 if tracks[i].state == 'Tracked']
    
    # 计算未匹配轨迹与低分框的 IoU 距离矩阵
    dists_second = iou_distance(r_tracked_stracks, det_low)
    # 再次使用匈牙利算法，放宽阈值（比如0.5）因为遮挡时框可能不太准
    matches_2, unmatched_tracks_2, unmatched_det_low = linear_assignment(dists_second, thresh=0.5)
    
    # 更新第二次（低分框）匹配成功的轨迹
    for trk_idx, det_idx in matches_2:
        r_tracked_stracks[trk_idx].update(det_low[det_idx])
        
    # --- 收尾工作 ---
    # 1. 两次都没匹配上的轨迹，标记为 Lost (后面连续几帧找不到就删除)
    for i in unmatched_tracks_2:
        r_tracked_stracks[i].mark_lost()
        
    # 2. 第一次没匹配上的高分框，认为是新出现的目标，初始化新轨迹
    for i in unmatched_det_high:
        self.init_track(det_high[i])
        
    # 3. 未匹配的低分框 (unmatched_det_low) 直接被丢弃，这就是噪音！
```

## 终极总结：工业界该如何选型？

经过从数学原理、流程图、再到具体执行细节的全面拆解，这两种经典范式的特点已完全浮出水面：

| 维度             | DeepSORT (特征级联派)                                        | ByteTrack (二次匹配派)                                       |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **核心信仰**     | “只要看清脸，跑多远我都能认回来”                             | “只要我不放弃模糊的线索，就不会丢”                           |
| **底层依赖**     | 卡尔曼 + 马氏距离 + 余弦距离(神经网络) + 匈牙利算法          | 仅靠 卡尔曼 + IoU(重叠度) + 匈牙利算法                       |
| **遮挡恢复能力** | **极强**。目标被完全遮挡长达数秒，再次出现只要长相没变，依然能找回原ID。 | **一般**。如果是“部分遮挡”表现完美；但如果“完全消失”几秒，由于没有外貌记忆，重新出现会变成新ID。 |
| **运行速度**     | **较慢**。受限于额外的 ReID 提取网络，通常只能跑到几十 FPS。 | **极快**。几乎没有任何额外开销，单纯的矩阵运算，轻松突破百 FPS。 |
| **场景建议**     | 跨镜头追踪、人流密集且衣服颜色各异的商场监控、对长时间遮挡要求高的场景。 | 算力受限的边缘设备（如自动驾驶终端）、高速公路车流统计、通用且对实时性要求极高的场景。 |

目标跟踪技术的演进，实际上就是一场**“如何更聪明地利用检测框和历史信息”**的博弈。理解了这套逻辑体系，你也就拿到了推开现代多目标跟踪大门的金钥匙。
